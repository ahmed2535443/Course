💡 مقدمة عن SEO والأداء

أهلاً بيكم يا صديقي في الدرس الثاني والعشرين! الدرس ده من أهم الدروس اللي هتأثر على نجاح متجرك الإلكتروني. ليه؟ لأن حتى لو متجرك فيه أحسن منتجات وأحسن تصميم، لو محدش لاقاه في جوجل، مفيش فايدة!

تخيل معايا: انت عملت متجر إلكتروني gorgeous فيه موبايلات بأسعار ممتازة. لكن لما حد يبحث في جوجل عن "شراء موبايل"، موقعك مش ظاهر في أول 10 نتائج. النتيجة؟ مفيش زوار ومفيش مبيعات.

🔍 لماذا SEO مهم للمتجر الإلكتروني؟

محركات البحث هي المصدر الرئيسي للزيارات للمتاجر الإلكترونية. بدون تحسين محركات البحث، ستفقد كثيراً من الزوار المحتملين. في هذا الدرس، هنتعلم كيف نحسن موقعنا ليظهر في أول نتائج البحث.

SEO اختصار لـ Search Engine Optimization. يعني ازاي ت optimise موقعك عشان يظهر في أول نتائج بحث جوجل. في المتاجر الإلكترونية، الـ SEO مهم جداً لأنه:

  • زيارات مجانية: بدل ما تدفع إعلانات، جوجل بيجيبلك زوار مجاناً.
  • ثقة العملاء: لما موقعك يظهر في أول نتيجة، العملاء بيثقوا فيه أكتر.
  • مبيعات أعلى: الزوار من جوجل بيبقوا ناويين يشتروا. يعني معدل التحويل أعلى.
  • نمو طويل المدى: الـ SEO بياخد وقت، لكن لما يشتغل، بيجيب نتائج طويلة.
💡 لماذا Next.js ممتاز لـ SEO؟
Next.js يوفر Server Side Rendering (SSR) و Static Site Generation (SSG) اللي بتساعد محركات البحث على فهم محتوى موقعك بشكل أفضل. كمان بيوفر prerendering للصفحات مما يحسن سرعة التحميل. محركات البحث بتحب المواقع السريعة والمحتوى الظاهر في HTML أول ما الصفحة تتفتح.

🏷️ إعداد Meta tags

في Next.js، يمكنك استخدام مكتبة next/head لإضافة meta tags لكل صفحة. لكن مع App Router، نستخدم Metadata object بدلاً من ذلك.

Meta tags دي معلومات بتظهر في كود الصفحة لكن مش بتظهر للمستخدم. بتساعد جوجل يفهم الصفحة ايه. أهم الـ meta tags هي:

  • title: عنوان الصفحة. بيظهر في تبويب المتصفح وفي نتائج البحث.
  • description: وصف الصفحة. بيظهر تحت العنوان في نتائج البحث.
  • keywords: الكلمات المفتاحية. بتساعد جوجل يفهم الصفحة عن ايه.
  • openGraph: معلومات لوسائل التواصل الاجتماعي.
  • twitter: معلومات لتويتر.
  • robots: تعليمات لجوجل عن الصفحة.
app/layout.js javascript 
export const metadata = {
  title: {
    default: 'Market - متجر إلكتروني',
    template: '%s | Market'
  },
  description: 'متجر إلكتروني متكامل بتصميم عصري وأداء عالي',
  keywords: ['متجر', 'تسوق', 'منتجات', 'إلكتروني'],
  authors: [{ name: 'Market Team' }],
  creator: 'Market',
  metadataBase: new URL('https://market.vercel.app'),
  openGraph: {
    type: 'website',
    locale: 'ar_SA',
    url: 'https://market.vercel.app',
    siteName: 'Market',
    title: 'Market - متجر إلكتروني',
    description: 'متجر إلكتروني متكامل بتصميم عصري وأداء عالي',
    images: [
      {
        url: '/og-image.jpg',
        width: 1200,
        height: 630,
        alt: 'Market - متجر إلكتروني'
      }
    ]
  },
  twitter: {
    card: 'summary_large_image',
    title: 'Market - متجر إلكتروني',
    description: 'متجر إلكتروني متكامل بتصميم عصري وأداء عالي',
    images: ['/og-image.jpg'],
    creator: '@market'
  },
  robots: {
    index: true,
    follow: true,
    googleBot: {
      index: true,
      follow: true,
      'max-video-preview': -1,
      'max-image-preview': 'large',
      'max-snippet': -1
    }
  }
};

شوف يا صديقي، الـ metadata object ده بيحط كل المعلومات اللي محتاجها في مكان واحد. شوف التفاصيل:

  • title.default: العنوان الأساسي للموقع كله.
  • title.template: الصيغة اللي هتتكرر في كل صفحة. يعني لو صفحة المنتج اسمها "موبايل آيفون"، العنوان الكامل هيبقى "موبايل آيفون | Market".
  • openGraph: المعلومات دي بتظهر لما حد يشارك رابط الموقع على فيسبوك أو لينكدإن. الصورة اللي حاططها (og-image.jpg) بتظهر مع الرابط.
  • robots: بتقول لجوجل "index: true" يعني اindex الصفحة، و "follow: true" يعني اتبع الروابط اللي فيها.
⚠️ غلطة شائعة
بعض المطورين بيحطوا keywords وهمية. جوجل مش بيستخدم keywords في الـ ranking، لكن ممكن يعاقبك لو كانوا مضللين. حط بس الكلمات اللي فعلاً مناسبة للصفحة.

تخصيص Meta tags لكل صفحة

يمكنك تخصيص Meta tags لكل صفحة على حدة باستخدام generateMetadata:

app/products/[id]/page.js javascript 
export async function generateMetadata({ params }) {
  const product = await getProduct(params.id);
  
  return {
    title: product.name,
    description: product.description,
    openGraph: {
      title: product.name,
      description: product.description,
      images: [product.image],
      type: 'product',
      price: {
        amount: product.price,
        currency: 'SAR'
      }
    },
    twitter: {
      card: 'summary_large_image',
      title: product.name,
      description: product.description,
      images: [product.image]
    }
  };
}

ده مهم جداً لكل منتج. لما حد يشارك رابط منتج على فيسبوك، الصورة والعنوان والسعر بتظهر تلقائياً. وده بيخلي المشاركة تكون احترافية وجذابة.

📱 Open Graph tags

Open Graph tags بتستخدم لتحسين مظهر الروابط لما يتم مشاركتها على وسائل التواصل الاجتماعي مثل Facebook و LinkedIn.

⚠️ ملاحظة مهمة
تأكد من أن صور Open Graph بحجم 1200x630 بكسل للحصول على أفضل عرض. استخدم صيغة JPG أو PNG بحجم أقل من 1MB. الصورة دي هي اللي بتظهر لما حد يشارك رابط موقعك.
next.config.js javascript 
/** @type {import('next').NextConfig} */
const nextConfig = {
  images: {
    remotePatterns: [
      {
        protocol: 'https',
        hostname: '**.supabase.co'
      },
      {
        protocol: 'https',
        hostname: '**.supabase.in'
      }
    ],
    formats: ['image/avif', 'image/webp'],
    minimumCacheTTL: 60 * 60 * 24 * 30
  },
  experimental: {
    optimizePackageImports: ['lucide-react', '@radix-ui/react-icons']
  }
};

module.exports = nextConfig;

شوف الإعدادات دي:

  • remotePatterns: بيسمح بتحميل الصور من Supabase. لو مобавتش الـ hostname، الصور مش هتتحمل.
  • formats: بيحسّن الصور لصيغة WebP أو AVIF. الصور دي أصغر بكتير من JPG/PNG.
  • minimumCacheTTL: الصور بتتخزن في الكاش لمدة 30 يوم. وده بيحسن السرعة.
  • optimizePackageImports: بيحسّن استيراد المكتبات الكبيرة زي lucide-react.

🖼️ تحسين الصور مع Next.js Image

مكون Image في Next.js بيوفر تحسينات تلقائية للصور مثل:

  • التحميل الكسول (Lazy Loading)
  • التحجيم التلقائي حسب حجم الشاشة
  • تحويل الصور لصيغة WebP أو AVIF
  • منع Cumulative Layout Shift (CLS)
💡 ايه هو Cumulative Layout Shift (CLS)؟
لما الصفحة بتتحمل، الصور بتأخد وقت. لو الصورة مش حاطط ليها حجم، الصفحة بتتحرك أول ما الصورة تتحمل. المستخدم بيرجع يضغط على زرار غلط. الـ Image component في Next.js بيحط حجم ثابت للصورة عشان الصفحة متتحركش.
components/products/ProductCard.jsx jsx 
import Image from 'next/image';

export default function ProductCard({ product }) {
  return (
    <div className="product-card">
      <div className="relative aspect-square">
        <Image
          src={product.image}
          alt={product.name}
          fill
          sizes="(max-width: 768px) 100vw, (max-width: 1200px) 50vw, 33vw"
          className="object-cover rounded-lg"
          priority={false}
          loading="lazy"
          placeholder="blur"
          blurDataURL="data:image/jpeg;base64,/9j/4AAQSkZJRg..."
        />
      </div>
      <div className="p-4">
        <h3 className="font-semibold text-lg mb-2">{product.name}</h3>
        <p className="text-primary font-bold">{product.price} ر.س</p>
      </div>
    </div>
  );
}

شوف الـ props المهمة في الـ Image component:

  • fill: بيخلي الصورة تملأ العنصر الأب. محتاج تحط العنصر الأب يكون relative.
  • sizes: بيحدد حجم الصورة على كل شاشة. على الموبايل (768px)، الصورة بتاخد العرض كله. على التابلت (1200px)، بتاخد نص العرض. وعلى الديسكتوب، بتاخد ثلث العرض.
  • loading="lazy": التحميل الكسول. الصورة مش بتتحمل غير لما المستخدم يسكرول لغايتها.
  • placeholder="blur": بيعرض صورة ضبابية صغيرة قبل الصورة الحقيقية.
  • blurDataURL: الصورة الصغيرة الضبابية. بيخلي الصفحة تبقى أسرع.

استخدام priority للصور المهمة

للسماح بتحميل الصور بشكل مبكر (Above the fold)، استخدم priority:

components/Hero.jsx jsx 
import Image from 'next/image';

export default function Hero() {
  return (
    <section className="relative h-[600px]">
      <Image
        src="/hero-bg.jpg"
        alt="خلفية الصفحة الرئيسية"
        fill
        className="object-cover"
        priority
        quality={85}
      />
      <div className="absolute inset-0 bg-black/50" />
      <div className="relative z-10 container mx-auto px-4">
        <h1 className="text-5xl font-bold text-white mb-4">
          اكتشف أحدث المنتجات
        </h1>
        <p className="text-xl text-gray-200 mb-8">
          تشكيلة واسعة من المنتجات عالية الجودة
        </p>
      </div>
    </section>
  );
}

الـ priority بيخلي الصورة تتحمل بدري (preconnect). مفيد للصور اللي في أعلى الصفحة (above the fold). وده بيخلي الصفحة تفتح أسرع.

💡 امتى تستخدم priority؟
استخدمه بس للصور اللي في أعلى الصفحة. الصور اللي المستخدم بيشوفها أول ما يفتح الصفحة. الصور اللي في الأسفل مش محتاجة priority لأنهم بيتتحملوا كسلول (lazy load).

✂️ Code Splitting التلقائي

Next.js يقوم بـ Code Splitting تلقائي لكل صفحة. هذا يعني أن المستخدم يحمّل فقط الكود المطلوب للصفحة الحالية.

✅ ميزة Code Splitting
مع Next.js، لا حاجة لإعداد Code Splitting يدوياً. كل صفحة في مجلد app تتحول إلى chunk منفصل يُحمّل عند الحاجة. يعني لو المستخدم فتح الصفحة الرئيسية، بيتحمّل الكود الخاص بالصفحة الرئيسية بس. ولما يروح صفحة المنتجات، بيتحمّل كود الصفحة دي.
Dynamic Import jsx 
// استيراد ديناميكي لمكون ثقيل
import dynamic from 'next/dynamic';

const HeavyChart = dynamic(() => import('@/components/HeavyChart'), {
  loading: () => <p>جاري التحميل...</p>,
  ssr: false
});

// استيراد ديناميكي بمكونات مشروطة
const AdminDashboard = dynamic(
  () => import('@/components/AdminDashboard'),
  { ssr: false }
);

// استخدام loading state
const ProductModal = dynamic(
  () => import('@/components/ProductModal'),
  {
    loading: () => <div className="modal-loading"><Spinner /></div>,
    ssr: false
  }
);

الـ dynamic import مهم جداً للمكونات الكبيرة. شوف الأنواع المختلفة:

  • HeavyChart: مكون رسم بياني كبير. بيتحمل بس لما المستخدم يفتح الصفحة. وده بيثقل الصفحة لو اتحمل من الأول.
  • AdminDashboard: مكون لوحة التحكم. محتاج بس لو المستخدم admin. فبنحمله ديناميكياً.
  • ProductModal: مودال (نافذة منبثقة). بيتحمل بس لما المستخدم يضغط على منتج. وده بيحسن سرعة الصفحة.

ssr: false معناها إن المكون ده مش بيشتغل على السيرفر. بيشتغل بس على العميل (client). ده مهم للمكونات اللي بتستخدم useState أو useEffect.

⚡ أداء Server Components

Server Components في Next.js بتشتغل على السيرفر فقط، مما يحسن الأداء بشكل كبير:

1. تقليل حجم JavaScript المرسل للعميل

Server Component (الافتراضي) jsx 
// هذا Server Component - لا يُرسل JavaScript للعميل
import { getProducts } from '@/services/products';

export default async function ProductList() {
  // يتم تنفيذ هذا الكود على السيرفر فقط
  const products = await getProducts();
  
  return (
    <div className="grid grid-cols-1 md:grid-cols-3 gap-6">
      {products.map(product => (
        <ProductCard key={product.id} product={product} />
      ))}
    </div>
  );
}

ده Server Component. شوف الفرق:

  • مفيش "use client": ده معناه إنه Server Component.
  • async function: ممكن نستخدم await عشان نجلب بيانات من السيرفر.
  • مفيش JavaScript للعميل: الكود ده بيشتغل على السيرفر بس. المستخدم مش بيحمل أي JavaScript.
  • أمان: ممكن نكتب كود يوصل قاعدة البيانات من غير ما نخاف إن المستخدم يشوف المفاتيح السرية.

2. Client Component عند الحاجة فقط

Client Component jsx 
"use client";

import { useState } from 'react';

export default function AddToCartButton({ productId }) {
  const [loading, setLoading] = useState(false);
  
  const handleAddToCart = async () => {
    setLoading(true);
    // إضافة للسلة
    await addToCart(productId);
    setLoading(false);
  };
  
  return (
    <button
      onClick={handleAddToCart}
      disabled={loading}
      className="btn-primary"
    >
      {loading ? 'جاري الإضافة...' : 'أضف للسلة'}
    </button>
  );
}

ده Client Component. شوف الفرق:

  • "use client": ده بيقول لـ Next.js إن المكون ده بيشتغل على العميل.
  • useState: محتاج hook، فلازم يكون Client Component.
  • onClick: محتاج أحداث تفاعلية، فلازم يكون Client Component.
⚠️ قاعدة مهمة
لا تستخدم "use client" إلا عند الضرورة. كل ما تحط "use client"، الكود بتاعك بيترسل للعميل. حاول تخلي أغلب المكونات Server Components وتحط "use client" بس للمكونات اللي فعلاً محتاجة تفاعل مع المستخدم.

📊 قياس الأداء

استخدم أدوات مثل Lighthouse و Web Vitals لقياس أداء موقعك:

المقياس الهدف الشرح
LCP < 2.5 ثانية Largest Contentful Paint - وقت تحميل أكبر عنصر
FID < 100 مللي ثانية First Input Delay - وقت استجابة أول تفاعل
CLS < 0.1 Cumulative Layout Shift - تغيرات التخطيط
TTFB < 800 مللي ثانية Time to First Byte - وقت استجابة السيرفر
💡 ازاي تقيس الأداء؟
  • Lighthouse: أداة في Chrome DevTools. افتح Developer Tools وروح تبويب Lighthouse.
  • PageSpeed Insights: أداة جوجل أونلاين. ادخل pagespeed.web.dev.
  • Web Vitals Extension: إضافة للمتصفح بتقيس Web Vitals.
  • Vercel Analytics: بيقيس أداء الموقع في بيئة الإنتاج.

🛠️ أفضل الممارسات

⚠️ تجنب هذه الأخطاء
  • لا تستخدم "use client" إلا عند الضرورة (عند الحاجة للـ hooks)
  • لا تُصدّر بيانات حساسة من Server Components
  • لا تستخدم useEffect لجلب البيانات في Server Components
  • تأكد من أن الصور بحجم مناسب قبل الرفع
  • لا تحمل مكتبات كبيرة في كل مكان
  • استخدم Dynamic Import للمكونات الثقيلة

1. تحسين قاعدة البيانات

services/products.js javascript 
import { cache } from 'react';

// استخدام cache لتجنب الطلبات المتكررة
export const getProducts = cache(async (filters = {}) => {
  const { data, error } = await supabase
    .from('products')
    .select('*')
    .order('created_at', { ascending: false });
  
  if (error) throw error;
  return data;
});

// استخدام Revalidation
export const revalidate = 3600; // تحديث كل ساعة

شوف هنا عملنا حاجتين مهمتين:

  • cache: لو نفس الطلب اتعمل أكتر من مرة في نفس الصفحة، بيتexecute مرة واحدة بس وبنخزن النتيجة.
  • revalidate: بيقول لـ Next.js يحدّث البيانات كل ساعة. يعني لو منتج جديد اتضاف، أقصى وقت هيتظهر خلاله هو ساعة.

2. تحسين التخزين المؤقت

app/api/products/route.js javascript 
import { NextResponse } from 'next/server';

export async function GET(request) {
  const products = await getProducts();
  
  return NextResponse.json(products, {
    headers: {
      'Cache-Control': 'public, s-maxage=3600, stale-while-revalidate=86400'
    }
  });
}

الـ Cache-Control header مهم جداً:

  • public: المحتوى يتخزن في أي كاش (CDN).
  • s-maxage=3600: الكاش بيفضل صالح لمدة ساعة.
  • stale-while-revalidate=86400: بعد ما الكاش يخلص، استخدم الكاش القديم وأنا بحدّث. وده بيخلي الموقع يفضل سريع حتى لو البيانات بتتجدد.

3. تحسين التحميل الأولي

app/layout.js javascript 
import { Inter } from 'next/font/google';

const inter = Inter({
  subsets: ['latin'],
  display: 'swap',
  preload: true
});

export default function RootLayout({ children }) {
  return (
    <html lang="ar" dir="ltr" className={inter.className}>
      <body>{children}</body>
    </html>
  );
}

الخطوة دي بتحسّن تحميل الخطوط. شوف:

  • display: 'swap': بيستخدم الخط العادي أول ما الصفحة تتفتح وبعدين بيحول للخط الجديد. وده بيخلي الصفحة تفتح أسرع.
  • preload: true: بيحمل ملف الخط بدري. وده بيقلل وقت التحميل.
