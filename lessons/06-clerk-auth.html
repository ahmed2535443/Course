نظام تسجيل الدخول (Clerk)

⏱️ 55 دقيقة 📖 عملي الوحدة 2: تأسيس المشروع

مقدمة مبسّطة عن المصادقة

قبل ما نبدأ في Clerk، خلّينا نفهم يعني إيه "تسجيل دخول" أصلاً.

💡 يعني إيه Authentication؟

تخيل إن عندك بيت. لما حد يجي ي knocking على الباب، بتعرفه إزاي؟ بتعرفه بـ "الوش" أو "المفتاح" أو "البطاقة الشخصية". Authentication هو نفس الفكرة - لما حد يجي على موقعك، لازم يثبت إنه هو فعلاً اللي بيقول إنه هو.

Authentication و Authorization: الفرق بيناتهم

📋 شرح مبسّط:

  • Authentication (المصادقة) = مين إنت؟ = بتثبت هويتك (زي لما تدخل بطاقتك الشخصية)
  • Authorization (التفويض) = إيه اللي تقدر تعمله؟ = مسموحلك تعمل إيه (زي لما تدخل البيت عندك بس، مش عند غيرك)

مثال: لما بتسجل دخول في Gmail، Authentication بيثبت إنك أحمد. Authorization بيخلي أحمد يشوف إيميلاته بس، مش إيميلات غيره.

ليه محتاجين نظام تسجيل دخول؟

  • حماية البيانات - كل عميل يشوف طلباته بس
  • تجربة مستخدم أفضل - العملاء يقدروا يرجعوا يكملوا طلباتهم
  • أمان المعاملات - ما حدش يقدر يطلب باسم غيره
  • بيانات شخصية - كل عميل عنده عنوان وتاريخ شراء

ليه Clerk بالذات؟

فيه كتير من حلول تسجيل الدخول زي Firebase Auth و Auth0 و Supabase Auth... ليه اختارنا Clerk؟

✅ مميزات Clerk للمبتدئين:

  • مجاني بالكامل - في خطة مجانية كافية للمشاريع الصغيرة (حتى 10,000 مستخدم)
  • واجهة جاهزة - صفحات تسجيل الدخول والتسجيل جاهزة، مش محتاج تصممها
  • زر المستخدم (UserButton) - زر يظهر صورة المستخدم وقائمة خيارات
  • حماية Middleware - سهل تحميل صفحات معينة
  • يعمل مع كل البرامج - Next.js, React, Vue, PHP...
  • دعم كامل للعربية - واجهة عربية
  • أسهل من Firebase - محتاج أقل كود

شرح Clerk Interface

إنشاء حساب Clerk خطوة بخطوة

الخطوة 1: فتح الموقع

افتح المتصفح واكتب https://clerk.com واضغط Enter.

هتشوف الصفحة الرئيسية. اضغط على زر "Get Started" (أو "ابدأ").

الخطوة 2: التسجيل

ممكن تسجل بـ:

  • GitHub (الأسهل والأسرع)
  • Google
  • البريد الإلكتروني

الخطوة 3: إنشاء أول مشروع

بعد التسجيل، Clerk هيعرض عليك إنشاء مشروع جديد. اضغط "Create Application".

الخطوة 4: اختيار Framework

اختار Next.js من القائمة. Clerk هيسألك عن:

  • Application name: اكتب اسم تطبيقك (مثلاً: market-store)
  • Choose sign-up methods: اختار البريد الإلكتروني + كلمة السر

الخطوة 5: حفظ بيانات الـ API

بعد الإنشاء، Clerk هيديك مفتاحين مهمين:

  • Publishable Key: مفتاح عام. يبدأ بـ pk_test_. تقدر تحطه في كود الـ Frontend.
  • Secret Key: مفتاح سري. يبدأ بـ sk_test_. ما تحطهش في كود الـ Frontend!

⚠️ مفتاح سري!

الـ Secret Key ده زي كلمة سر البنك بتاعك. لو حد شافه، يقدر يتحكم في حسابك بالكامل. ما تحطهش في أي مكان في كود الـ Frontend، واحفظه في ملف .env.local بس.

شرح API Keys

📋 يعني إيه API Key؟

تخيل الـ API Key زي "بطاقة الدخول" بتاعت موقعك. لما موقعك يكلم Clerk، بيبعتله البطاقة دي عشان Clerk يعرفه ويوافق على الطلبات.

أنواع الـ Keys

  • Publishable Key (pk_test_...): دي عامة. تحطها في ملف .env.local والـ Frontend يقدر يشوفها.
  • Secret Key (sk_test_...): دي سرية. تحطها في .env.local بس. الـ Frontend ما يشوفهاش أبداً.

💡 الفرق بين Environment Variables العادية و الخاصة بـ Clerk:

  • اللي تبدأ بـ NEXT_PUBLIC_: الـ Frontend يقدر يشوفها (زي Publishable Key)
  • اللي ما تبدأش بـ NEXT_PUBLIC_: الـ Backend بس يقدر يشوفها (زي Secret Key)

تثبيت Clerk

الخطوة 1: تثبيت الحزمة

افتح الـ Terminal واكتب:

npm install @clerk/nextjs

📋 يعني إيه @clerk/nextjs؟

ده الـ SDK بتاع Clerk للـ Next.js. ببساطة، ده مكتبة بتعمل كل الشغل الصعب اللي ورا تسجيل الدخول - الـ Forms، الـ Tokens، الـ Middleware، كل حاجة.

الخطوة 2: إضافة Environment Variables

افتح ملف .env.local وأضف:

NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY=pk_test_c3RhYmxlLXRvbWNhdC00My5jbGVyay5hY2NvdW50cy5kZXYk
CLERK_SECRET_KEY=sk_test_1KwIOWeX2uk1jFNl4ygUs9Rca2CPpS3KE76wNZsGp0

⚠️ ملاحظة مهمة!

لاحظ إن الـ Publishable Key تبدأ بـ NEXT_PUBLIC_ عشان الـ Frontend يقدر يستخدمها. الـ Secret Key ما تبدأش بـ NEXT_PUBLIC_ عشان تبقى خاصة بالـ Backend بس.

الخطوة 3: إنشاء ClerkProvider

ملف app/layout.js لازم يحتوي على ClerkProvider. خلّينا نشوف الكود:

import { ClerkProvider } from '@clerk/nextjs'

export default function RootLayout({ children }) {
  return (
    <html lang="ar" dir="rtl">
      <body>
        <ClerkProvider>
          {children}
        </ClerkProvider>
      </body>
    </html>
  )
}

📋 يعني إيه Provider ببساطة؟

تخيل الـ Provider زي "مفتاح الرئيسي" للمبنى. كل شقة (صفحة) في المبنى محتاجة المفتاح ده عشان تفتح. الـ ClerkProvider بيدّي كل الصفحات في التطبيق إمكانية استخدام Clerk.

إنشاء صفحات تسجيل الدخول

Clerk بيوفرلك صفحات جاهزة لتسجيل الدخول والتسجيل. كل اللي محتاجه إنك تعمل الملفات الصح.

إنشاء صفحة تسجيل الدخول

أنشئ ملف app/sign-in/[[...sign-in]]/page.js:

import { SignIn } from '@clerk/nextjs'

export default function SignInPage() {
  return (
    <div className="flex justify-center items-center min-h-screen">
      <SignIn />
    </div>
  )
}

📋 شرح الكود:

  • import { SignIn } from '@clerk/nextjs': بنبعت الـ SignIn component من Clerk. ده الـ Form الجاهز لتسجيل الدخول.
  • export default function SignInPage(): دي الصفحة اللي بتظهر لما المستخدم يروح على /sign-in.
  • <SignIn />: ده الـ Form الجاهز. فيه حقل الإيميل وكلمة السر وزر تسجيل الدخول.

إنشاء صفحة التسجيل

أنشئ ملف app/sign-up/[[...sign-up]]/page.js:

import { SignUp } from '@clerk/nextjs'

export default function SignUpPage() {
  return (
    <div className="flex justify-center items-center min-h-screen">
      <SignUp />
    </div>
  )
}

شرح [[...sign-in]] و [[...sign-up]]

📋 يعني إيه Catch-All Route؟

لما تكتب [[...sign-in]]، ده معناه "التقط كل المسارات الفرعية". يعني الصفحة دي هتشتغل مع أي رابط زي:

  • /sign-in
  • /sign-in/factor-one
  • /sign-in/factor-two
  • /sign-in/reset-password

Clerk محتاج الـ Catch-All Route عشان يقدر يتحكم في الخطوات المختلفة (التحقق من البريد، كلمة السر الجديدة، إلخ).

💡 الفرق بين [...] و [[...]]:

  • [...slug] = Required = لازم يكون فيه مسار (زي /sign-in/factor-one)
  • [[...slug]] = Optional = ممكن يكون فاضي (زي /sign-in بس)

Clerk محتاج [[...]] عشان أحياناً ما يكونش فيه مسار فرعي.

UserButton شرح تفصيلي

UserButton هو زر ذكي بيوفر:

  • صورة المستخدم
  • قائمة خيارات (إعدادات الحساب، تسجيل الخروج)

استخدامه في Header

import { UserButton } from '@clerk/nextjs'

export default function Header() {
  return (
    <header>
      <nav>
        <a href="/">الرئيسية</a>
        <a href="/products">المنتجات</a>
        <a href="/cart">السلة</a>
      </nav>
      
      <!-- UserButton بيظهر تلقائياً لو المستخدم مسجل دخول -->
      <UserButton afterSignOutUrl="/" />
    </header>
  )
}

📋 شرح UserButton:

  • بيظهر صورة المستخدم لو مسجل دخول
  • بيضغطه يفتح قائمة فيها: "Manage Account" و "Sign Out"
  • afterSignOutUrl="/": بعد تسجيل الخروج، يروح على الصفحة الرئيسية

تخصيص UserButton

<UserButton
  afterSignOutUrl="/"
  appearance={{
    elements: {
      avatarBox: "w-10 h-10",  // حجم الصورة
    },
  }}
/>

SignedIn و SignedOut

ده من أهم المفاهيم في Clerk. umiejęلك تتحكم في مين يشوف إيه.

📋 شرح مبسّط:

  • <SignedIn>: المحتوى اللي جواه يظهر بس لو المستخدم مسجل دخول
  • <SignedOut>: المحتوى اللي جواه يظهر بس لو المستخدم مش مسجل دخول

مثال: Header ذكي

import { SignedIn, SignedOut, UserButton } from '@clerk/nextjs'

export default function Header() {
  return (
    <header className="flex justify-between items-center p-4">
      <a href="/" className="text-xl font-bold">🛒 المتجر</a>

      <div className="flex items-center gap-4">
        <a href="/products">المنتجات</a>
        <a href="/cart">السلة</a>
        
        {/*
          لو المستخدم مسجل دخول: نورّي UserButton
          لو مش مسجل دخول: نورّي أزرار تسجيل الدخول والتسجيل
        */}
        <SignedIn>
          <UserButton afterSignOutUrl="/" />
        </SignedIn>
        
        <SignedOut>
          <a href="/sign-in" className="px-4 py-2 bg-blue-600 text-white rounded">
            تسجيل الدخول
          </a>
          <a href="/sign-up" className="px-4 py-2 bg-gray-200 rounded">
            إنشاء حساب
          </a>
        </SignedOut>
      </div>
    </header>
  )
}

useAuth hook شرح تفصيلي

الـ useAuth hook بيدّيك معلومات عن المستخدم الحالي.

📋 يعني إيه Hook ببساطة؟

تخيل الـ Hook زي "زر سحري" لما بتضغطه، بيدّيك معلومات معينة. الـ useAuth بيدّيك معلومات عن المستخدم: مين هو، هل مسجل دخول، إلخ.

مثال: التحقق من تسجيل الدخول

'use client'

import { useAuth } from '@clerk/nextjs'

export default function MyComponent() {
  const { userId, isLoaded, isSignedIn } = useAuth()

  // لسه بيحمل
  if (!isLoaded) {
    return <div>...جاري التحميل</div>
  }

  // مش مسجل دخول
  if (!isSignedIn) {
    return <div>لازم تسجل دخول الأول</div>
  }

  // مسجل دخول
  return (
    <div>
      <p>مرحباً! معرف المستخدم: {userId}</p>
    </div>
  )
}

شرح الحقول:

  • userId: المعرف الفريد للمستخدم
  • isLoaded: هل البيانات اتحملت؟ (لما الصفحة بتحل، بيكون false لثانية)
  • isSignedIn: هل المستخدم مسجل دخول؟ (true أو false)

💡 نصيحة مهمة:

دايماً تحقق من isLoaded قبل ما تحقق من أي حاجة تانية. لو ما تحققتش، ممكن تحصل مشاكل.

Middleware شرح مبسّط

الـ Middleware ده كود بيشتغل "قبل" ما الصفحة تتحمّل. بيحمي الصفحات.

💡 يعني إيه Middleware ببساطة؟

تخيل إن عندك بواب (حارس) قدام باب الشقة. كل ما حد يجي، البواب بيسأله "مين إنت؟". لو قال "أنا أحمد"، بيفتحله. لو ما قالش، بيرجعه برا. الـ Middleware هو البواب بتاع الموقع.

إنشاء Middleware

أنشئ ملف middleware.js في الـ Root بتاع المشروع (جنب package.json):

import { clerkMiddleware } from '@clerk/nextjs/server'

export default clerkMiddleware()

export const config = {
  matcher: [
    //تخطي الصفحات اللي مش محتاجين authentication
    '/((?!_next|[^?]*\\.(?:html?|css|js(?!on)|jpe?g|webp|png|gif|svg|ttf|woff2?|ico|csv|docx?|xlsx?|zip|webmanifest)).*)',
    '/(api|trpc)(.*)',
  ],
}

📋 شرح الكود:

  • import { clerkMiddleware } from '@clerk/nextjs/server': بنستورد الـ Clerk Middleware.
  • export default clerkMiddleware(): بنفعّل الـ Middleware.
  • config.matcher: بنحدد الصفحات اللي Middleware يشتغل عليها. الـ regex ده بيقاطع كل الصفحات ما عدا الصور والـ CSS والـ JS.

حماية صفحات معينة

لو عايز تحمي صفحة معينة، ممكن تستخدم protectedRoutes:

import { clerkMiddleware, createRouteMatcher } from '@clerk/nextjs/server'

// الصفحات المحمية - لازم المستخدم يكون مسجل دخول
const protectedRoutes = createRouteMatcher([
  '/dashboard(.*)',
  '/checkout(.*)',
  '/orders(.*)',
  '/wishlist(.*)',
  '/admin(.*)',
])

export default clerkMiddleware(async (auth, req) => {
  // لو الصفحة محمية، تحقق من تسجيل الدخول
  if (protectedRoutes(req)) {
    await auth.protect()
  }
})

export const config = {
  matcher: [
    '/((?!_next|[^?]*\\.(?:html?|css|js(?!on)|jpe?g|webp|png|gif|svg|ttf|woff2?|ico|csv|docx?|xlsx?|zip|webmanifest)).*)',
    '/(api|trpc)(.*)',
  ],
}

📋 شرح createRouteMatcher:

  • '/dashboard(.*)': كل الروابط اللي تبدأ بـ /dashboard (زي /dashboard/settings)
  • '/checkout(.*)': كل الروابط اللي تبدأ بـ /checkout
  • '/orders(.*)': كل الروابط اللي تبدأ بـ /orders

✅ إيه اللي بيحصل لما المستخدم يحاول يدخل صفحة محمية بدون تسجيل دخول؟

Clerk تلقائياً بيرجعه على صفحة /sign-in. بعد ما يسجل دخول، بيرجعه على الصفحة الأصلية تلقائياً!

الاتصال بين Clerk و Supabase

دلوقتي عندنا Clerk للتسجيل الدخول و Supabase للبيانات. محتاجين نربطهم ببعض.

💡 يعني إيه ربطهم؟

لما المستخدم يسجل دخول عبر Clerk، محتاجين نعرف مين هو في Supabase عشان نحفظ طلباته وبياناته. Clerk بيدينا userId وده نفس الرقم اللي بنستخدمه في Supabase.

خطوة 1: استخدام userId من Clerk في Supabase

import { useAuth } from '@clerk/nextjs'
import { supabase } from '@/lib/supabase'

export default function CheckoutPage() {
  const { userId } = useAuth()

  async function handleCheckout(shippingData) {
    // حفظ الطلب في Supabase مع userId بتاع Clerk
    const { error } = await supabase
      .from('orders')
      .insert({
        user_id: userId,  // معرف المستخدم من Clerk
        items: cartItems,
        total: cartTotal,
        ...shippingData
      })

    if (!error) {
      // نجاح!
    }
  }
}

خطوة 2: جلب طلبات المستخدم

import { useAuth } from '@clerk/nextjs'
import { supabase } from '@/lib/supabase'
import { useEffect, useState } from 'react'

export default function OrderHistory() {
  const { userId } = useAuth()
  const [orders, setOrders] = useState([])

  useEffect(() => {
    async function fetchOrders() {
      const { data } = await supabase
        .from('orders')
        .select('*')
        .eq('user_id', userId)  // بس الطلبات بتاعتي
        .order('created_at', { ascending: false })
      
      setOrders(data || [])
    }

    if (userId) {
      fetchOrders()
    }
  }, [userId])

  return (
    <div>
      {orders.map(order => (
        <div key={order.id}>
          طلب رقم: {order.id}
          الحالة: {order.status}
        </div>
      ))}
    </div>
  )
}

استكشاف الأخطاء

مشكلة 1: "ClerkProvider is not wrapping the app"

⚠️ السبب:

الـ ClerkProvider مش موجود في ملف app/layout.js.

✅ الحل:

تأكد إن <ClerkProvider> موجود في ملف الـ Root Layout وبيلف كل الـ children.

مشكلة 2: "Invalid publishable key"

⚠️ السبب:

الـ Publishable Key غلط أو مش موجود.

✅ الحل:

تأكد إنك حطيت المفتاح الصح في .env.local وإنك عملت restart للـ Server.

مشكلة 3: الصفحة مش بتتحمّل بعد تسجيل الدخول

⚠️ السبب:

الـ Middleware مش موجود أو فيه خطأ فيه.

✅ الحل:

تأكد إن ملف middleware.js موجود في الـ Root بتاع المشروع (جنب package.json).

مشكلة 4: "You're not signed in"

⚠️ السبب:

الصفحة محمية بـ Middleware والمستخدم مش مسجل دخول.

✅ الحل:

تأكد إن الصفحة مش موجودة في الـ protectedRoutes لو مش عايز تحميها. أو سجّل دخول الأول.

مشكلة 5: صور المستخدم مش بتظهر

⚠️ السبب:

المستخدم ما رفعش صورة个人ية.

✅ الحل:

ده عادي. Clerk بياخد صورة افتراضية من الأول حرف الاسم.
