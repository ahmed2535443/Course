حماية صفحات Admin ونظام الصلاحيات

⏱️ 35 دقيقة 📖 عملي الوحدة 2: قاعدة البيانات والمصادقة

🎯 أهداف الدرس

أهلاً بيك في الدرس السابع من وحدة قاعدة البيانات والمصادقة! في الدرس ده هنتعلم حاجة مهمة جداً لأمان أي موقع إلكتروني - إزاي نحمي صفحات معينة عشان مش أي حد يقدر يدخلها.

تخيّل إنك بتبني متجر إلكتروني. فيه صفحات عامة زي صفحة المنتجات اللي أي زبون يقدر يشوفها، وفيه صفحات خاصة زي لوحة التحكم (Dashboard) اللي بس صاحب المتجر يقدر يدخلها ويعدّل في المنتجات والأسعار والطلبات. لو ما حميتهاش، أي حد يقدر يفتحها ويعدّل في كل حاجة!

  • إيه هي حماية الصفحات وليه مهمة جداً
  • إيه هو الـ Middleware وازاي بيشتغل
  • إزاي نكتب Middleware يحمي صفحات Admin
  • إزاي نتحقق من صلاحيات المستخدم (Admin ولا مستخدم عادي)
  • إزاي نعمل صفحة Login ونعيد التوجيه بعد تسجيل الدخول
  • إزاي نحمي API Routes من الوصول غير المصرح به
  • الأخطاء الشائعة اللي Beginners بيقعوا فيها
💡 ملاحظة مهمة
الدرس ده مبني على اللي اتعلمناه في الدرس السادس (Clerk Authentication). لو ما خلصتش الدرس ده الأول، رجّع وخلّصه الأول عشان فهم الدرس ده يكون أسهل.

🛡️ يعني إيه Middleware؟

خلّيني أشرحلك حاجة مهمة جداً قبل ما نبدأ. الـ Middleware في Next.js هو كود بيشتغل قبل ما الطلب يوصل للصفحة اللي المستخدم عايز يفتحها. يعني لو حد زار رابط زي /admin/dashboard، الـ Middleware بيشتغل الأول وبيقرر: أسيبه يدخل ولا لا.

تشبيه بسيط - الحارس على الباب 🚪

تخيّل إن عندك مبنى وفيه حارس على الباب. كل ما حد بييجي عشان يدخل المبنى، الحارس بيسأله:

  1. "مين أنت؟" - يعني مين الشخص ده (ده بيقابل إنك تسأل عن هوية المستخدم)
  2. "عندك تصريح؟" - يعني هل عندك حق تدخل المبنى ده (ده بيقابل إنك تتحقق من الصلاحيات)
  3. "روّح!" أو "ادخل!" - الحارس بيقرر يسيبك تدخل ولا يرجّعك (ده بيقابل الـ Redirect)

الـ Middleware في Next.js بيشتغل بنفس الطريقة بالظبط. هو بيشتغل على السيرفر (مش في المتصفح بتاع المستخدم)، وبيراقب كل طلب بيجي للموقع، وبيقرر يسيبه يوصل للصفحة ولا لا.

ليه Middleware أهم من الحماية في الصفحة نفسها؟

ممكن تقول: "ليه ما أعملش حماية جوا الصفحة نفسها؟" السبب إن الـ Middleware بيشتغل قبل ما الصفحة تتحمل أصلاً. لو حطيت الحماية جوا الصفحة، الصفحة هتتحمل الأول وبعدين هتتحقق - وده معناه إن الكود بتاع الصفحة هيشتغل حتى لو المستخدم ما عنهوش صلاحية. الـ Middleware بيوفر عليك ده كله وبيوقف الطلب من الأول.

⚠️ فرق مهم
Middleware بيشتغل على السيرفر (Server Side) وبي前瞻性 الطلب قبل ما يوصل للصفحة.
protects في الصفحة بيشتغل في المتصفح (Client Side) وبيرجع بعد ما الصفحة اتحملت.
Middleware هو الحل الأقوى والأمان.

استخدامات Middleware في Next.js:

  • حماية المسارات - منع المستخدمين من الوصول لصفحات محددة زي صفحات Admin
  • إعادة التوجيه - لو المستخدم ما عندوش صلاحية، نوديه على صفحة تسجيل الدخول
  • إضافة Headers - إضافة معلومات إضافية للطلب زي التوكن أو اللغة
  • التحقق من الصلاحيات - التأكد إن المستخدم هو Admin ولا مستخدم عادي
  • إعادة كتابة الروابط - غير شكل الرابط من غير ما يتغير في شريط العنوان
  • فحص اللغة أو المنطقة - إظهار محتوى مختلف حسب لغة المستخدم أو بلده

🔧 شرح middleware.js بالتفصيل

دلوقتي هنشوف الكود بالتفصيل ونشرح كل سطر فيه. الخطوة الأولى هي إنشاء ملف middleware.js في جذر المشروع.

⚠️ تنبيه مهم جداً
ملف middleware.js لازم يكون في جذر المشروع (بجانب مجلد app) ومش جوا مجلد app. لو حطيته في مجلد غلط، الـ Middleware مش هيشتغل خالص.

هيكل المجلدات الصحيح:

📁 market/
├── 📄 middleware.js # هنا الـ Middleware (جذر المشروع)
├── 📄 package.json
├── 📄 next.config.js
├── 📁 app/
│ ├── 📁 (admin)/ # صفحات Admin
│ ├── 📁 (shop)/ # صفحات المتجر
│ ├── 📁 (auth)/ # صفحات تسجيل الدخول
│ └── 📄 layout.js
├── 📁 components/
└── 📁 lib/

شرح الكود سطر بسطر:

middleware.js javascript 
// السطر الأول والثاني: بنستورد الأدوات اللي هنحتاجها من Clerk
import { clerkMiddleware, createRouteMatcher } from '@clerk/nextjs/server';

السطر الأول والثاني: بنستورد دالتين من Clerk. الدالة الأولى clerkMiddleware دي الـ Middleware الرئيسي بتاع Clerk اللي بيعمل المصادقة. الدالة التانية createRouteMatcher دي بتساعدنا نحدد المسارات (الروابط) اللي عايزين نحميها بشكل سهل.

middleware.js - تعريف المسارات العامة javascript 
// تعريف المسارات العامة (ممكن أي حد يوصلها من غير تسجيل دخول)
const publicPaths = createRouteMatcher([
  '/',                    // الصفحة الرئيسية
  '/products(.*)',        // صفحة المنتجات + أي صفحة فرعية جواها
  '/about',               // صفحة عن المتجر
  '/contact',             // صفحة التواصل
  '/sign-in(.*)',         // صفحة تسجيل الدخول + أي صفحة فرعية
  '/sign-up(.*)',         // صفحة التسجيل + أي صفحة فرعية
  '/api(.*)',             // كل API Routes
]);

شرح الكود: هنا بنقول للـ Middleware: "الروابط دي مسموح لأي حد يدخلها من غير ما نتحقق". يعني لو حد فتح الصفحة الرئيسية أو صفحة المنتجات أو صفحة التسجيل، الـ Middleware بسيبه يدخل من غير ما يسأله عن أي حاجة.

العلامة (.*): دي معناها "أي حاجة بعدها". يعني /products(.*) معناها الصفحة الرئيسية للمنتجات + أي صفحة فرعية جواها زي /products/123 أو /products/shoes.

middleware.js - تعريف مسارات Admin javascript 
// تعريف مسارات Admin (محتاجين صلاحيات Admin عشان ندخلها)
const adminPaths = createRouteMatcher([
  '/admin(.*)',              // كل الروابط اللي تبدأ بـ /admin
  '/admin/dashboard(.*)',    // لوحة التحكم الرئيسية
  '/admin/products(.*)',     // إدارة المنتجات
  '/admin/orders(.*)',       // إدارة الطلبات
]);

شرح الكود: هنا بنحدد الروابط اللي عايزينها تكون محمية. أي رابط يبدأ بـ /admin لازم المستخدم يكون مسجل دخول وهو Admin عشان يقدر يدخلها.

middleware.js - الـ Middleware الرئيسي javascript 
// الـ Middleware الرئيسي - ده الكود اللي بيشتغل على كل طلب
export default clerkMiddleware(async (auth, req) => {
  
  // الخطوة الأولى: لو المسار عام، اسمح بالوصول من غير أي تحقق
  if (publicPaths(req)) {
    return;  // هنا بنقول: "خلاص، مفيش حاجة نعملها، سيبه يدخل"
  }
  
  // الخطوة التانية: لو المسار تبع Admin، نتحقق من الصلاحيات
  if (adminPaths(req)) {
    
    // بنجيب معلومات المستخدم الحالي من Clerk
    const { userId } = await auth();
    
    // لو المستخدم ما سجلش دخول (userId فاضي)، نوديه على صفحة تسجيل الدخول
    if (!userId) {
      return Response.redirect(new URL('/sign-in', req.url));
    }
    
    // لو المستخدم مسجل دخول، نتحقق إنه Admin
    // هنستخدم البريد الإلكتروني في المثال ده
    const user = await auth.user?.();
    const isAdmin = user?.emailAddresses?.[0]?.emailAddress === 'admin@example.com';
    
    // لو المستخدم مش Admin، نوديه على الصفحة الرئيسية
    if (!isAdmin) {
      return Response.redirect(new URL('/', req.url));
    }
  }
  
  // الخطوة التالتة: لباقي المسارات، تحقق إن المستخدم مسجل دخول
  const { userId } = await auth();
  
  // لو المستخدم مش مسجل دخول، وده رابط محمي (مش عام)، نوديه على تسجيل الدخول
  if (!userId) {
    return Response.redirect(new URL('/sign-in', req.url));
  }
});

شرح تفصيلي:

  • clerkMiddleware(async (auth, req) => { ... }): ده الـ Middleware الرئيسي. الكود جواه بيشتغل على كل طلب بيجي للموقع.
  • auth: دي دالة بترجع معلومات المستخدم الحالي (زي الـ userId).
  • req: دي الطلب نفسه. فيها معلومات زي الرابط والـ Headers.
  • publicPaths(req): بنسأل: "الرابط ده من المسارات العامة؟" لو أيوه، بنسيبه يدخل.
  • adminPaths(req): بنسأل: "الرابط ده تبع Admin؟" لو أيوه، بنتحقق من الصلاحيات.
  • Response.redirect(): بتعيد توجيه المستخدم لرابط تاني.
middleware.js - config javascript 
// الـ config بيحدد على أي طلبات يشتغل الـ Middleware
export const config = {
  matcher: [
    // الخط الأول: بيتخطى كل الملفات الثابتة (صور، CSS، JS)
    '/((?!_next|[^?]*\\.(?:html?|css|js(?!on)|jpe?g|webp|png|gif|svg|ttf|woff2?|ico|csv|docx?|xlsx?|zip|webmanifest)).*)',
    
    // الخط التاني: بيشتغل على كل API Routes
    '/(api|trpc)(.*)',
  ],
};

شرح الكود: الـ config.matcher ده مهم جداً. هو بيقول للـ Middleware: "اشتغل بس على الطلبات دي". الخط الأول معناه: "استبعد كل الملفات الثابتة زي الصور والـ CSS والـ JS، عشان ما نعملش حماية على الملفات دي". الخط التاني معناه: "كمان اشتغل على كل API Routes".

✅ ملخص شرح middleware.js
  1. بنستورد أدوات Clerk
  2. بنحدد المسارات العامة (مسموح لأي حد)
  3. بنحدد مسارات Admin (محتاج صلاحيات)
  4. بنكتب الـ Middleware اللي بيتحقق من كل طلب
  5. بنحدد على أي طلبات يشتغل الـ Middleware

📊 حماية صفحة Dashboard - خطوة بخطوة

دلوقتي هنشوف إزاي نحمي صفحة Dashboard. الخطوات هي:

الخطوة 1: إنشاء مجلد صفحات Admin

أول حاجة نعملها هي إنشاء مجلد اسمه (admin) جوا مجلد app. الـ (admin) دي Route Group في Next.js، يعني بتساعدنا ننظم الصفحات من غير ما تضيف رابط إضافي.

📁 app/
├── 📁 (admin)/ # صفحات Admin
│ ├── 📄 layout.js # Layout خاص بـ Admin
│ ├── 📄 page.js # Dashboard (الصفحة الرئيسية)
│ ├── 📁 products/
│ │ └── 📄 page.js # إدارة المنتجات
│ └── 📁 orders/
│ └── 📄 page.js # إدارة الطلبات
├── 📁 (shop)/ # صفحات المتجر
├── 📁 (auth)/ # صفحات Auth
├── 📄 layout.js # Layout رئيسي
└── 📄 page.js
💡 يعني إيه Route Group؟
في Next.js، لما تحط اسم المجلد في أقواس زي (admin)، ده معناه إنه Route Group. الـ Route Group بيساعدك تنظم الكود من غير ما يغير في الروابط. يعني الرابط هيفضل /admin مش /(admin).

الخطوة 2: إنشاء Layout خاص بـ Admin

الـ Layout ده هيشتغل على كل الصفحات جوا مجلد (admin). هو اللي هيعرض الشريط الجانبي والقائمة.

app/(admin)/layout.js javascript 
"use client";

import { useAuth, useUser } from '@clerk/nextjs';
import { useRouter } from 'next/navigation';
import { useEffect } from 'react';
import Link from 'next/link';

export default function AdminLayout({ children }) {
  // بنجيب معلومات المستخدم من Clerk
  const { userId } = useAuth();    // الـ ID بتاع المستخدم
  const { user } = useUser();       // كل المعلومات بتاعته
  const router = useRouter();       // للتنقل بين الصفحات
  
  // التحقق من إن المستخدم Admin
  useEffect(() => {
    // لو المستخدم مسجل دخول
    if (userId) {
      // بنتحقق من إن المستخدم Admin
      // هنا بنستخدم البريد الإلكتروني كمثال بسيط
      // في المشاريع الحقيقية، بتتحقق من قاعدة البيانات
      const isAdmin = user?.emailAddresses?.[0]?.emailAddress === 'admin@example.com';
      
      // لو مش Admin، نوديه على الصفحة الرئيسية
      if (!isAdmin) {
        router.push('/');
      }
    }
  }, [userId, user, router]);  // الـ useEffect بيشتغل لما البيانات دي تتغير
  
  // لو المستخدم ما سجلش دخول، نعرض رسالة تحميل
  if (!userId) {
    return (
      <div className="flex items-center justify-center min-h-screen">
        <p>جاري التحقق من الصلاحيات...</p>
      </div>
    );
  }
  
  // لو كل حاجة تمام، نعرض الـ Layout
  return (
    <div className="flex min-h-screen">
      {/* الشريط الجانبي */}
      <aside className="w-64 bg-gray-900 text-white p-4">
        <div className="mb-8">
          <h2 className="text-xl font-bold">لوحة التحكم</h2>
          <p className="text-sm text-gray-400">مرحباً {user?.firstName}</p>
        </div>
        
        {/* قائمة الروابط */}
        <nav className="space-y-2">
          <Link href="/admin" className="block p-2 rounded hover:bg-gray-800">
            Dashboard
          </Link>
          <Link href="/admin/products" className="block p-2 rounded hover:bg-gray-800">
            المنتجات
          </Link>
          <Link href="/admin/orders" className="block p-2 rounded hover:bg-gray-800">
            الطلبات
          </Link>
        </nav>
      </aside>
      
      {/* المحتوى الرئيسي */}
      <main className="flex-1 p-8">
        {children}
      </main>
    </div>
  );
}

شرح الكود:

  • "use client": بنقول إن الملف ده Client Component عشان نستخدم useEffect وـ useRouter.
  • useAuth(): بترجع معلومات تسجيل الدخول (زي الـ userId).
  • useUser(): بترجع كل معلومات المستخدم (الاسم، البريد الإلكتروني، الصورة).
  • useEffect: دالة بتشتغل لما الصفحة تتحمل أو لما البيانات تتغير. هنا بنستخدمها نتحقق من إن المستخدم Admin.
  • router.push('/'): بنودي المستخدم على الصفحة الرئيسية لو مش Admin.
  • {children}: هنا بيتحط المحتوى بتاع الصفحة اللي جوا الـ Layout (زي صفحة Dashboard نفسها).

🔑 التحقق من صلاحيات Admin

في الأقسام اللي فاتت، استخدمنا البريد الإلكتروني عشان نتحقق إن المستخدم Admin. لكن ده مش الحل الأفضل في المشاريع الحقيقية. خليني أشرحلك الطرق المختلفة:

الطريقة 1: التحقق من البريد الإلكتروني (مبسطة)

دي أبسط طريقة بس مش مناسبة للمشاريع الحقيقية لأنك لو غيرت البريد الإلكتروني بتاع الـ Admin، لازم تغير الكود كمان.

التحقق من البريد الإلكتروني javascript 
// في middleware.js
const user = await auth.user?.();
const isAdmin = user?.emailAddresses?.[0]?.emailAddress === 'admin@example.com';

if (!isAdmin) {
  return Response.redirect(new URL('/', req.url));
}

الطريقة 2: التحقق من قاعدة البيانات (المحسّنة)

دي الطريقة الصح. بتخزن معلومات الـ Admin في قاعدة البيانات، وبتحقق منها في الـ Middleware.

التحقق من قاعدة البيانات javascript 
// في ملف منفصل (مثلاً lib/checkAdmin.js)
import { supabase } from '@/lib/supabase';

export async function checkIsAdmin(userId) {
  // بنبحث عن المستخدم في جدول المستخدمين
  const { data, error } = await supabase
    .from('users')               // جدول المستخدمين
    .select('role')              // بنجيب بس عمود الـ role
    .eq('clerk_id', userId)      // بنبحث بالـ clerk_id
    .single();                   // بنجيب نتيجة واحدة بس

  // لو في مشكلة أو المستخدم مش موجود
  if (error || !data) {
    return false;
  }

  // بنتحقق إن الـ role هو admin
  return data.role === 'admin';
}
استخدام الدالة في middleware.js javascript 
import { clerkMiddleware, createRouteMatcher } from '@clerk/nextjs/server';
import { checkIsAdmin } from '@/lib/checkAdmin';

const publicPaths = createRouteMatcher([
  '/',
  '/products(.*)',
  '/sign-in(.*)',
  '/sign-up(.*)',
]);

const adminPaths = createRouteMatcher([
  '/admin(.*)',
]);

export default clerkMiddleware(async (auth, req) => {
  if (publicPaths(req)) {
    return;
  }
  
  if (adminPaths(req)) {
    const { userId } = await auth();
    
    if (!userId) {
      return Response.redirect(new URL('/sign-in', req.url));
    }
    
    // بنتحقق من قاعدة البيانات لو المستخدم Admin
    const isAdmin = await checkIsAdmin(userId);
    
    if (!isAdmin) {
      return Response.redirect(new URL('/', req.url));
    }
  }
  
  const { userId } = await auth();
  if (!userId) {
    return Response.redirect(new URL('/sign-in', req.url));
  }
});

🔄 redirect_url شرح وتفصيل

دي من أهم الميزات في تجربة المستخدم. لما المستخدم يحاول يفتح صفحة محمية زي /admin/dashboard وهو ما سجلش دخول، بنوديه على صفحة تسجيل الدخول. لكن السؤال: بعد التسجيل، يروح فين؟

لو ما استخدمناش redirect_url، المستخدم هيرجع الصفحة الرئيسية بدل ما يرجع للصفحة اللي كان عايز يفتحها. وده بيضايق المستخدم.

المشكلة بدون redirect_url:

  1. المستخدم يضغط على /admin/dashboard
  2. الـ Middleware يوديه على /sign-in
  3. المستخدم يسجّل دخول
  4. يرجع الصفحة الرئيسية (الصفحة اللي كان عايزها ضاعت!)

الحل بـ redirect_url:

  1. المستخدم يضغط على /admin/dashboard
  2. الـ Middleware يحفظ الرابط ويردّيه على /sign-in?redirect_url=/admin/dashboard
  3. المستخدم يسجّل دخول
  4. Clerk يوديه تلقائياً على /admin/dashboard
middleware.js مع redirect_url javascript 
export default clerkMiddleware(async (auth, req) => {
  const { userId } = await auth();
  
  // لو المسار عام، اسمح بالوصول
  if (publicPaths(req)) {
    // لو المستخدم مسجل دخول وحاول يفتح صفحة تسجيل الدخول
    // نوديه الصفحة الرئيسية (مش محتاج يسجّل دخول تاني)
    if (userId && (
      req.nextUrl.pathname === '/sign-in' || 
      req.nextUrl.pathname === '/sign-up'
    )) {
      return Response.redirect(new URL('/', req.url));
    }
    return;
  }
  
  // لو المسار تبع Admin
  if (adminPaths(req)) {
    if (!userId) {
      // بنحفظ الرابط الأصلي في redirect_url
      const signInUrl = new URL('/sign-in', req.url);
      signInUrl.searchParams.set('redirect_url', req.url);
      // النتيجة: /sign-in?redirect_url=http://localhost:3000/admin/dashboard
      return Response.redirect(signInUrl);
    }
    
    // تحقق من الصلاحيات هنا
    const isAdmin = await checkIsAdmin(userId);
    if (!isAdmin) {
      return Response.redirect(new URL('/', req.url));
    }
  }
  
  // لباقي المسارات المحمية
  if (!userId) {
    return Response.redirect(new URL('/sign-in', req.url));
  }
});
💡 إزاي Clerk بيتعامل مع redirect_url
Clerk بيفهم تلقائياً إن في redirect_url في الرابط، وبعد ما المستخدم يسجّل دخول، بيردّيه على الرابط الأصلي. أنت مش محتاج تعمل حاجة إضافية - Clerk بيعملها لوحده!

🔒 حماية API Routes

مفيش فايدة من حماية الصفحات لو الـ API Routes محميةش كمان. حد ممكن يعرف رابط الـ API من الكود ويستخدمه من غير ما يسجّل دخول.

app/api/admin/products/route.js javascript 
import { auth } from '@clerk/nextjs/server';
import { NextResponse } from 'next/server';
import { supabase } from '@/lib/supabase';
import { checkIsAdmin } from '@/lib/checkAdmin';

// GET: جلب كل المنتجات (Admin فقط)
export async function GET() {
  // بنتحقق من المستخدم أولاً
  const { userId } = await auth();
  
  // لو ما فيش مستخدم مسجل دخول
  if (!userId) {
    return NextResponse.json(
      { error: 'غير مصرح - لازم تسجّل دخول' },
      { status: 401 }  // 401 معناه Unauthorized
    );
  }
  
  // بنتحقق إن المستخدم Admin
  const isAdmin = await checkIsAdmin(userId);
  if (!isAdmin) {
    return NextResponse.json(
      { error: 'غير مصرح - لازم تكون Admin' },
      { status: 403 }  // 403 معناه Forbidden
    );
  }
  
  // لو كل حاجة تمام، بنجيب المنتجات
  const { data, error } = await supabase
    .from('products')
    .select('*')
    .order('created_at', { ascending: false });
  
  // لو في مشكلة في قاعدة البيانات
  if (error) {
    return NextResponse.json(
      { error: 'مشكلة في جلب البيانات' },
      { status: 500 }  // 500 معناه Internal Server Error
    );
  }
  
  // بنرجع المنتجات
  return NextResponse.json(data);
}

// POST: إضافة منتج جديد (Admin فقط)
export async function POST(request) {
  const { userId } = await auth();
  
  if (!userId) {
    return NextResponse.json(
      { error: 'غير مصرح' },
      { status: 401 }
    );
  }
  
  const isAdmin = await checkIsAdmin(userId);
  if (!isAdmin) {
    return NextResponse.json(
      { error: 'غير مصرح - لازم تكون Admin' },
      { status: 403 }
    );
  }
  
  // بنجيب البيانات اللي بعتها المستخدم
  const body = await request.json();
  
  // بنضيف المنتج في قاعدة البيانات
  const { data, error } = await supabase
    .from('products')
    .insert([body])
    .select();
  
  if (error) {
    return NextResponse.json(
      { error: 'مشكلة في إضافة المنتج' },
      { status: 500 }
    );
  }
  
  // بنرجع المنتج اللي اتنضاف
  return NextResponse.json(data[0], { status: 201 });  // 201 معناه Created
}

شرح أكواد الحالة (Status Codes):

الكود الاسم المعنى
200 OK نجاح - كل حاجة تمام
201 Created تم إنشاء حاجة جديدة بنجاح
401 Unauthorized غير مصرح - لازم تسجّل دخول
403 Forbidden ممنوع - عندك دخول بس ما عندكش صلاحية
404 Not Found مش موجود
500 Internal Server Error مشكلة في السيرفر

❌ الأخطاء الشائعة عند Beginners

ده قسم مهم جداً. الأخطاء دي وقعت فيها ناس كتير، فانت انتبه ليها من الأول:

❌ الخطأ 1: حط middleware.js في مجلد غلط
الغلط: حط الملف في مجلد app/ أو مجلد تاني.
الصحيح: لازم يكون في جذر المشروع بجانب package.json.
العلامة: لو شايف app/ بجانب الملف، يبقى صح.
❌ الخطأ 2: نسيان صفحة sign-in في publicPaths
الغلط: ما حطيتش /sign-in(.*) في المسارات العامة.
النتيجة: المستخدم ما يقدرش يفتح صفحة تسجيل الدخول لأن الـ Middleware بيردده!
الحل: دايماً حط صفحات Auth في publicPaths.
❌ الخطأ 3: إعادة توجيه لا نهائية
الغلط: الصفحة بتتعمل reload باستمرار أو بتدي خطأ "loop".
السبب: الـ Middleware بيردّي المستخدم على صفحة Auth، والصفحة بتوديه تاني على Auth، وهكذا.
الحل: تأكد إن صفحات Auth في publicPaths وإن ما فيش شرط بيردّي على Auth وهو في Auth.
❌ الخطأ 4: حماية Client Side بس
الغلط: حطيت الحماية في الصفحة نفسها بـ useEffect من غير Middleware.
المشكلة: المستخدم ممكن يوصل للبيانات من غير ما الصفحة تتحمل (مثلاً من API).
الحل: لازم Middleware يحمي على مستوى السيرفر + حماية إضافية في الصفحة.
❌ الخطأ 5: نسيان حماية API Routes
الغلط: حميت الصفحات بس ونسيت الـ API Routes.
المشكلة: حد ممكن يعرف رابط الـ API من الـ Network tab في المتصفح ويستخدمه من غير تسجيل دخول.
الحل: دايماً تحقق من auth() في كل API Route.
❌ الخطأ 6: استخدام التحقق من البريد الإلكتروني بس
الغلط: بتتحقق إن المستخدم Admin بالبريد الإلكتروني بس.
المشكلة: لو غيرت البريد الإلكتروني بتاع الـ Admin، لازم تغير الكود كمان. وده مش عملي.
الحل: استخدم قاعدة البيانات وحدد صلاحيات المستخدمين في جدول مستقل.

📊 مقارنة أنواع الحماية

فيه طرق كتير تحمي بيها صفحاتك. خليني أشرحلك الفرق بينهم:

الطريقة بتحمي إزاي الاستخدام مميزاتها
Middleware على السيرفر قبل ما الصفحة تتحمل حماية كل المسارات بناءً على الشرط أقوى طريقة، بتشتغل قبل ما أي كود يتنفذ
useAuth() في المتصفح بعد ما الصفحة تتحمل إظهار/إخفاء المحتوى في الصفحة سهلة، بس مش أقوى حاجة
auth() على السيرفر في Server Components جلب بيانات المستخدم من قاعدة البيانات amenable للـ Server Components
SignedIn/SignedOut في المتصفح إظهار محتوى مختلف حسب حالة تسجيل الدخول سهلة جداً وواضحة
API Route Protection في الـ API Route نفسه حماية Endpoints من الوصول غير المصرح به مهم للـ APIs اللي بتتعامل مع البيانات
💡 أيه الأفضل؟
الأفضل تستخدم كلهم مع بعض! Middleware للحماية الأساسية، useAuth() للمحتوى في الصفحة، وحماية الـ API Routes للبيانات. كل طريقة بتكمل التانية.
