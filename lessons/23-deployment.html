🚀 مقدمة عن النشر

أهلاً بيكم يا صديقي في الدرس التالت والعشرين! لحد كده انت بنيت متجرك الإلكتروني وكملته. لكن المشكلة إن الموقع لسه شغال بس على جهازك (localhost:3000). عشان أي حد يقدر يستخدمه، لازم تنشره على الإنترنت.

النشر (Deployment) هو عملية نقل المشروع من جهازك الشخصي لسيرفر على الإنترنت. السيرفر ده شغال 24 ساعة و7 أيام في الأسبوع ومتصل بالإنترنت. الناس بتقدر تفتح موقعك من أي مكان في العالم.

💡 تخيّل معايا
تخيّل إنك بنيت محل تجاري جميل في بيتك. المحل شكله حلو ومليان منتجات. بس المشكلة إنه في بيتك ومفيش حد يعرفه. النشر هو إنك تنقل المحل لشارع رئيسي الناس كلها بتعدّ عليه. كده الناس هتقدر تدخل وتشتري من عندك.

🔄 الفرق بين التطوير والإنتاج

قبل ما ننشر، لازم نفهم الفرق بين بيئة التطوير وبيئة الإنتاج. الفرق ده مهم جداً عشان كتير من الأخطاء بعد النشر سببها إن الناس مش فاهمة الفرق.

بيئة التطوير (Development)

  • Hot Reload: تحديث فوري للمتغيرات من غير refresh
  • أخطاء تفصيلية: browser بيوريك كل التفاصيل
  • بدون بناء: المشروع بيشتغل من غير build
  • Debugging سهل: console و debugger متاحة

بيئة الإنتاج (Production)

  • سرعة عالية: المشروع بيتعمله build مسبقاً
  • أخطاء مختفية: المستخدم مش هيشوف أخطاء تفصيلية
  • ملفات مضغوطة: CSS و JavaScript بيتم ضغطهم
  • أمان أعلى: بعض الأشياء بتتعمل بشكل مختلف
الميزة التطوير الإنتاج
الرابط localhost:3000 your-domain.com
الأمر npm run dev npm run build
السرعة عادية سريعة جداً
الأخطاء تفصيلية مخفية
⚠️ غلطة شائعة
كتير من المبتدئين بيجربوا يفتحوا localhost من موبايلهم ويقولوا "نشرت!". ده مش نشر حقيقي. النشر لازم يتعمل build و deploy على سيرفر.

⭐ لماذا Vercel؟

Vercel هو أفضل منصة لنشر مشاريع Next.js. ليه؟

  • مجاني: الخطة المجانية بتكفي للمشاريع الشخصية
  • تلقائي: كل ما تعمل push، الموقع بيتحدث تلقائياً
  • سريع: CDN عالمي بيشتغل في أي دولة
  • سهل: النشر بخطوات بسيطة
  • متوافق مع Next.js: Vercel صاحب Next.js نفسه
  • SSL مجاني: شهادة SSL تلقائية لكل موقع
  • Analytics مجانية: أدوات مراقبة الأداء
💡 مقارنة سريعة
ممكن تنشر على Netlify أو Railway أو DigitalOcean. لكن Vercel هو الأفضل لـ Next.js لأنه بيقدم ميزات زي Edge Functions و Image Optimization بشكل تلقائي.

📦 إنشاء حساب GitHub

GitHub هو أكبر مستودع لتخزين الكود المصدري. قبل النشر لازم يكون عندك حساب عليه.

خطوات إنشاء الحساب

  1. افتح github.com واضغط Sign up
  2. ادخل بريدك الإلكتروني واعمل كلمة مرور قوية
  3. اختار اسم مستخدم: مثلاً ahmed-market-dev
  4. أكد حسابك: هيوصلك إيميل من GitHub اضغط رابط التأكيد

تثبيت Git

تحميل وتثبيت Git bash 
# حمّل Git من: https://git-scm.com/download/win
# بعد التثبيت، تأكد:
git --version
# المفروض يطلع: git version 2.45.0

إعداد بيانات Git

إعداد Git bash 
git config --global user.name "Ahmed Hassan"
git config --global user.email "ahmed@example.com"
# لازم الإيميل يكون نفس إيميل GitHub

⬆️ رفع المشروع على GitHub

إنشاء مستودع على GitHub

  1. ادخل على GitHub واضغط + ثم New repository
  2. اكتب اسم المشروع (مثلاً market-store)
  3. اختار Private للمشروع الشخصي
  4. متختارش README أو .gitignore (احنا هنعملها)
  5. اضغط Create repository

رفع الكود

رفع المشروع على GitHub bash 
# 1. افتح التيرمينال في مجلد المشروع
cd "E:\تطبيقات veb coding\market"

# 2. ابدأ مستودع Git
git init

# 3. أضف كل الملفات
git add .

# 4. اعمل commit
git commit -m "المشروع جاهز للنشر"

# 5. اربط المشروع بـ GitHub
git remote add origin https://github.com/ahmed-market-dev/market-store.git

# 6. ارفع الكود
git push -u origin main

شرح الأوامر:

  • git init: بيلغي إنشاء مستودع Git جديد
  • git add .: بيضيف كل الملفات للمستودع
  • git commit: بيخزن التغييرات مع رسالة وصفية
  • git remote add: بيربط المشروع المحلي بـ GitHub
  • git push: بيرفع الكود على GitHub

ملف .gitignore

.gitignore text 
# الملفات اللي لازم تتخطى
.next/
node_modules/
.env
.env.local
.DS_Store
.vscode/
⚠️ خطر أمني
لو رفعت .env.local على GitHub بال случай، المفاتيح السرية بتاعتك هتكون ظاهرة للعالم كله. لو حصل كده، ا立马 اعمل rotate لكل المفاتيح.

🔐 إعداد Environment Variables

المتغيرات البيئية بتحتوي على معلومات سرية مش عايزها تظهر في الكود.

لماذا نستخدمها؟

لو حطيت المفاتيح السرية في الكود ورفعته على GitHub، أي حد يقدر يسرقها. عشان كده بنخفيها في ملفات منفصلة.

.env.local bash 
# Clerk Authentication
NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY=pk_test_...
CLERK_SECRET_KEY=sk_test_...
NEXT_PUBLIC_CLERK_SIGN_IN_URL=/sign-in
NEXT_PUBLIC_CLERK_SIGN_UP_URL=/sign-up
NEXT_PUBLIC_CLERK_AFTER_SIGN_IN_URL=/
NEXT_PUBLIC_CLERK_AFTER_SIGN_UP_URL=/

# Supabase
NEXT_PUBLIC_SUPABASE_URL=https://your-project.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=eyJ...
SUPABASE_SERVICE_ROLE_KEY=eyJ...

# Site URL
NEXT_PUBLIC_SITE_URL=http://localhost:3000

شرح المتغيرات

  • NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY: المفتاح العام. بيبدأ بـ pk_
  • CLERK_SECRET_KEY: المفتاح السري. بيبدأ بـ sk_. لا تشاركه!
  • NEXT_PUBLIC_SUPABASE_URL: رابط مشروع Supabase
  • NEXT_PUBLIC_SUPABASE_ANON_KEY: المفتاح العام لـ Supabase
  • SUPABASE_SERVICE_ROLE_KEY: المفتاح السري. للسيرفر بس
  • NEXT_PUBLIC_SITE_URL: رابط الموقع. في الإنتاج يتغير لرابط Vercel
💡 NEXT_PUBLIC وليه مش؟
المتغيرات اللي بتبدأ بـ NEXT_PUBLIC متاحة للعميل. مش ممكن تحط فيها مفاتيح سرية. اللي مش بتبدأ بـ NEXT_PUBLIC بتشتغل بس على السيرفر.

إضافة المتغيرات على Vercel

  1. افتح Vercel Dashboard واختر مشروعك
  2. اذهب إلى Settings ثم Environment Variables
  3. أضف كل متغير بشكل منفصل
  4. اختار البيئات (Production و Preview)
  5. اضغط Save لكل متغير
⚠️ غلطة شائعة
بعض المطورين بيحطوا المتغيرات في Production بس. لازم تحطها في Preview كمان عشان Preview Deployments تشتغل صح.

🗄️ إعداد Supabase للإنتاج

قائمة مراجعة Supabase

✅ قبل النشر
  • ✅ تفعيل Row Level Security (RLS) على كل جدول
  • ✅ كتابة Policies مناسبة لكل جدول
  • ✅ مراجعة Database Functions
  • ✅ إنشاء Backup من القاعدة
  • ✅ تأكد المفتاح العام محدود الصلاحيات
تفعيل RLS وكتابة Policies sql 
-- تفعيل RLS على جدول المنتجات
ALTER TABLE products ENABLE ROW LEVEL SECURITY;

-- السماح بالقراءة للجميع
CREATE POLICY "Products are viewable by everyone"
ON products FOR SELECT USING (true);

-- السماح بالكتابة للمشرفين بس
CREATE POLICY "Only admins can insert products"
ON products FOR INSERT
WITH CHECK (auth.uid() IN (
  SELECT user_id FROM admin_users
));

👤 إعداد Clerk للإنتاج

خطوات الإعداد

  1. افتح Clerk Dashboard واختر مشروعك
  2. اذهب إلى Configure ثم Paths
  3. أضف رابط موقعك في Allowed Redirect URLs
  4. أضف: https://market-store.vercel.app/**
  5. لو عندك Domain مخصص، أضفه كمان
Redirect URLs المطلوبة text 
# للتطوير المحلي
http://localhost:3000/**

# لبيئة Vercel
https://market-store.vercel.app/**

# للنطاق المخصص
https://market.com/**

🌐 اسم Domain مخصص

بعد النشر، موقعك هيشتغل على market-store.vercel.app. لو عايز اسم احترافي، محتاج Domain مخصص.

خطوات شراء Domain

  1. اختار اسم Domain مناسب (قصير وسهل)
  2. البحث عن.Domain في Namecheap أو GoDaddy
  3. شراء Domain (حوالي 10-15 دولار/سنة)

نصائح لاختار Domain

  • قصيرة: shop.com أحسن من myamazingonlinestore.com
  • .com هو الأفضل: لو متاح، ده الأشهر
  • متعلق بالنشاط: مثال: alnoor-books.com

ربط Domain بـ Vercel

  1. افتح Vercel Dashboard → مشروعك → Settings → Domains
  2. أضف Domain (مثلاً: market.com)
  3. استنسخ DNS Records من Vercel
  4. روح على لوحة تحكم Domain وعدّل الـ DNS
DNS Records المطلوبة text 
# للنطاق الرئيسي
Type: A
Name: @
Value: 76.76.21.21

# للنطاق الفرعي (www)
Type: CNAME
Name: www
Value: cname.vercel-dns.com

التفعيل بيأخذ من 24 لـ 48 ساعة. استن لحد ما يخلص.

⚠️ تحديث بعد Domain
بعد تفعيل Domain، حدّث NEXT_PUBLIC_SITE_URL في Vercel وحدّث Redirect URLs في Clerk Dashboard.

🔒 شهادة SSL و HTTPS

HTTPS هو بروتوكول أمن بيشتغل عليه كل المواقع المحترمة. لما بتفتح موقع وتشوف قفل أخضر، ده معناه إن الاتصال مشفّر.

ليه HTTPS مهم؟

  • الأمان: بيشفّر كل البيانات
  • SEO: Google بتحب المواقع الآمنة
  • الثقة: المستخدمين بيثقوا في المواقع الآمنة
  • المتاجر: ضروري لقبول بطاقات الائتمان

الخبر الحلو: Vercel بيوفر SSL مجاني تلقائياً. مش محتاج تعمل أي حاجة.

💡 تأكد من HTTPS
افتح الموقع في المتصفح وشوف شريط العنوان. لو فيه قفل أخضر أو "Secure"، يبقى HTTPS شغال. أو اكتب في Terminal: curl -I https://your-site.vercel.app

❌ أخطاء شائعة في النشر

الخطأ 1: Environment Variables ناقصة

السبب: المتغيرات مش مضافة على Vercel.
الحل: تأكد إن كل المتغيرات موجودة في Settings → Environment Variables. وتأكد إن الأسماء متطابقة بالظبط.

الخطأ 2: نسيان عمل Build

السبب: المشروع شغال على localhost بس.
الحل: جرّب npm run build على الكمبيوتر. لو في أخطاء، اتصلحها قبل النشر.

الخطأ 3: رفع .env.local على GitHub

السبب: الملف مش موجود في .gitignore.
الحل: احذف الملف من GitHub فوراً وعمل rotate لكل المفاتيح.

الخطأ 4: الروابط الثابتة

السبب: استخدام localhost:3000 في الكود.
الحل: استخدم process.env.NEXT_PUBLIC_SITE_URL بدل الروابط الثابتة.

الخطأ 5: مشاكل الصور

السبب: الصور مش محملة من HTTPS أو domains مش مضافة في next.config.js.
الحل: تأكد إن الصور من HTTPS واضف الـ domains في next.config.js.

الخطأ 6: نسيان Redirect URLs في Clerk

السبب: Clerk مش عارف يرجع المستخدم.
الحل: أضف رابط موقعك في Clerk Dashboard → Configure → Paths → Redirect URLs.

الخطأ 7: Route Handlers مش شغالة

السبب: في اختلافات بين dev و production في معالجة الطلبات.
الحل: جرّب تعمل npm run build وشوف لو في أخطاء. وتأكد إن الـ Route Handlers مكتوبة صح.

الخطأ 8: Server Components و Client Components

السبب: استخدام 'use client' في أماكن مش محتاجها أو استخدام Server Features في Client Component.
الحل: راجع الكود وتأكد إن كل Component في مكانه الصح.

💡 نصيحة للتجنب
قبل ما تنشر، اعمل npm run build على جهازك. لو البناء نجح من غير أخطاء، يبقى المشروع جاهز للنشر. لو في أخطاء، اتصلحها الأول.

✅ قائمة ما بعد النشر

فحص عام

  • ✅ تأكد إن الصور كلها بتتحمل
  • ✅ تأكد إن الروابط الداخلية كلها شغالة

النشر التلقائي

من أعظم ميزات Vercel: كل ما تعمل git push، الموقع بيتحدث تلقائياً. مش محتاج تعمل أي حاجة.

النشر التلقائي bash 
git add .
git commit -m "إصلاح خطأ في صفحة المنتجات"
git push
# Vercel هتبدأ تبني وتنشر تلقائياً
💡 Preview Deployments
لو بتشتغل مع فريق أو بتعمل Pull Request، Vercel بينشر نسخة تجريبية. كده تقدر تراجع التعديلات قبل الدمج.

مراقبة الأداء بعد النشر

بعد النشر، لازم تراقب أداء موقعك بشكل مستمر:

  • Vercel Analytics: راقب Web Vitals (LCP, FID, CLS)
  • Google Search Console: راقب SEO و#indexed pages
  • Sentry: راقب الأخطاء في Production
فحص سريع من Terminal bash 
# فحص سرعة الموقع
curl -o /dev/null -s -w "%{time_total}\n" https://your-site.vercel.app

# فحص SSL
echo | openssl s_client -connect your-site.vercel.app:443 -servername your-site.vercel.app 2>/dev/null | openssl x509 -noout -dates
