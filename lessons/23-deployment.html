💡 مقدمة عن النشر

أهلاً بيكم يا صديقي في الدرس التالت والعشرين! لحد كده انت بنيت متجرك الإلكتروني وكملته. لكن المشكلة إن الموقع لسه شغال بس على جهازك (localhost:3000). عشان أي حد يقدر يستخدمه، لازم تنشره على الإنترنت.

النشر (Deployment) هو عملية نقل المشروع من جهازك الشخصي لسيرفر على الإنترنت. وVercel هو أفضل منصة لنشر مشاريع Next.js. ليه؟ لأنه:

  • مجاني للمشاريع الشخصية: مش محتاج تدفع حاجة.
  • تلقائي: كل ما تعمل push للكود، الموقع بيتنشر تلقائياً.
  • سريع جداً: بيوفر CDN عالمي. يعني الموقع بيشتغل بسرعة في أي دولة.
  • سهل: النشر بخطوات بسيطة.
  • متوافق مع Next.js: Vercel صاحب Next.js. فبيفهمه أحسن من أي حد تاني.
💡 Vercel vs منصات تانية
ممكن تنشر على Netlify أو Railway أو حتى سيرفرك الشخصي. لكن Vercel هو الأفضل لـ Next.js لأنه بيقدم ميزات زي Edge Functions و Image Optimization و Analytics بشكل تلقائي.

📋 قائمة ما قبل النشر

قبل نشر المشروع، تأكد من إتمام هذه الخطوات:

💡 قائمة مراجعة ما قبل النشر
  • ✅ اختبار جميع الميزات بشكل كامل
  • ✅ مراجعة جميع الأخطاء والتنبيهات
  • ✅ تحسين أداء الصور
  • ✅ مراجعة Meta tags و SEO
  • ✅ اختبار النماذج ومداخل المستخدم
  • ✅ التأكد من عمل نظام تسجيل الدخول
  • ✅ اختبار نظام الدفع (إن وجد)
  • ✅ مراجعة الأمان

القائمة دي مهمة جداً. قبل ما تنشر، لازم تتأكد إن كل حاجة شغالة. لأن بعد النشر، أي خطأ هيظهر لكل المستخدمين. فخلّيني أشرحلك كل نقطة:

  • اختبار جميع الميزات: افتح كل صفحة واضغط كل زرار. جرّب تسجل دخول وخروج. جرّب تشتري منتج. جرّب تضيف منتج للسلة والمفضلة.
  • مراجعة الأخطاء: افتح Console في المتصفح وشوف لو في أخطاء حمراء. لو في، لازم تصلحها قبل النشر.
  • تحسين الصور: تأكد إن الصور مش كبيرة جداً. الصور الكبيرة بتبثّي الصفحة.
  • Meta tags: تأكد إن كل صفحة عندها title و description.
  • النماذج: جرّب تبعت كل نموذج. جرّب تبعت نموذج فاضي. جرّب تبعت بيانات غلط.
  • تسجيل الدخول: جرّب تسجل دخول وخروج. جرّب تسجل دخول بكلمة مرور غلط.
  • الأمان: تأكد إنه مفيش مفاتيح API ظاهرة في الكود. تأكد إن صفحة Admin محمية.

1. تنظيف المشروع

تنظيف الملفات غير المستخدمة bash 
# حذف الملفات المؤقتة
rm -rf .next
rm -rf node_modules/.cache

# تثبيت الاعتماديات
npm install

# بناء المشروع للتأكد من عدم وجود أخطاء
npm run build

# تنظيف ملفات التطوير
npm cache clean --force

خطوات التنظيف دي مهمة:

  • rm -rf .next: بيحذف ملفات البناء القديمة. ممكن تكون فيها ملفات مش محدثة.
  • rm -rf node_modules/.cache: بيحذف الكاش بتاع الاعتماديات.
  • npm install: بيثبّت الاعتماديات من جديد. ممكن يكون فيه تحديثات مهمة.
  • npm run build: بيبني المشروع. لو في أي خطأ، هيتكتب هنا قبل النشر.
  • npm cache clean --force: بيحذف الكاش العام.
⚠️ غلطة شائعة
بعض المطورين بيعملوا npm run build قبل كل نشر. ده مهم جداً لأنه بيكشف الأخطاء اللي مش بتظهر في npm run dev. لأن في اختلافات بين بيئة التطوير وبيئة الإنتاج.

تأكد إن الـ scripts المهمة موجودة:

  • start: لتشغيل المشروع بعد البناء.
  • lint: لفحص الكود وتنقيفه.

🔐 إعداد متغيرات البيئة

المتغيرات البيئية (Environment Variables) ضرورية لتخزين المفاتيح السرية. في Next.js، نستخدم ملف .env.local للتطوير المحلي.

⚠️ تحذير أمني
لا ترفع ملف .env.local إلى GitHub أو أي مستودع عام. تأكد من وجوده في ملف .gitignore. لو المفاتيح السرية ظهرت públicas، ممكن حد يستخدمها ويخترق موقعك.
.env.local bash 
# Clerk Authentication
NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY=pk_test_...
CLERK_SECRET_KEY=sk_test_...
NEXT_PUBLIC_CLERK_SIGN_IN_URL=/sign-in
NEXT_PUBLIC_CLERK_SIGN_UP_URL=/sign-up
NEXT_PUBLIC_CLERK_AFTER_SIGN_IN_URL=/
NEXT_PUBLIC_CLERK_AFTER_SIGN_UP_URL=/

# Supabase
NEXT_PUBLIC_SUPABASE_URL=https://your-project.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=eyJ...
SUPABASE_SERVICE_ROLE_KEY=eyJ...

# Site URL
NEXT_PUBLIC_SITE_URL=http://localhost:3000

# Stripe (optional)
STRIPE_SECRET_KEY=sk_test_...
NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY=pk_test_...
STRIPE_WEBHOOK_SECRET=whsec_...

شوف يا صديقي، كل متغير ليه دور:

  • NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY: مفتاح Clerk العام. بيبدأ بـ pk_.
  • CLERK_SECRET_KEY: مفتاح Clerk السري. بيبدأ بـ sk_. لا تشاركه مع حد!
  • NEXT_PUBLIC_SUPABASE_URL: رابط مشروع Supabase بتاعك.
  • NEXT_PUBLIC_SUPABASE_ANON_KEY: مفتاح Supabase العام.
  • SUPABASE_SERVICE_ROLE_KEY: مفتاح Supabase السري. بيستخدم في السيرفر بس.
  • NEXT_PUBLIC_SITE_URL: رابط الموقع. في الإنتاج، لازم يتغير لرابط Vercel.
💡 ليه في NEXT_PUBLIC ومش NEXT_PUBLIC؟
المتغيرات اللي بتبدأ بـ NEXT_PUBLIC متاحة للعميل (المستخدم). يعني ممكن تظهر في المتصفح. المتغيرات اللي مش بتبدأ بـ NEXT_PUBLIC بتشتغل بس على السيرفر. يعني السر مش هيتعرض.

إعداد Environment Variables على Vercel

بعد رفع المشروع على Vercel، يجب إضافة المتغيرات البيئية من خلال لوحة التحكم:

  1. افتح لوحة تحكم Vercel واختر مشروعك
  2. اذهب إلى Settings ثم Environment Variables
  3. أضف كل متغير بشكل منفصل مع القيم الصحيحة
  4. اختر البيئات (Production, Preview, Development)
  5. اضغط Save لحفظ كل متغير
⚠️ غلطة شائعة
بعض المطورين بيحطوا متغيرات البيئة في Production بس. لازم تحطها في Preview كمان عشانhiênات النشر المسبقة (Preview Deployments) تشتغل صح.

🚀 نشر المشروع على Vercel

Vercel هو أفضل منصة لنشر مشاريع Next.js. إليك الخطوات:

الطريقة الأولى: من خلال واجهة Vercel

خطوات النشر text 
1. اذهب إلى vercel.com وسجل دخول بحسابك
2. اضغط "New Project"
3. اختر "Import Git Repository"
4. اختر مستودع GitHub الخاص بك
5. اضغط "Import"
6. تأكد من إعدادات البناء:
   - Framework Preset: Next.js
   - Build Command: npm run build
   - Output Directory: .next
7. أضف Environment Variables
8. اضغط "Deploy"

الخطوات دي بسيطة جداً. انت بتعمل:

  • تسجيل دخول: استخدم حساب GitHub بتاعك.
  • New Project: مشروع جديد.
  • Import: استيراد المشروع من GitHub.
  • Settings: تأكد إن الإعدادات صحيحة.
  • Deploy: اضغط وانتظر. النشر بياخد تقريباً دقيقة واحدة.

الطريقة الثانية: من خلال CLI

تثبيت ونشر عبر CLI bash 
# تثبيت Vercel CLI
npm install -g vercel

# تسجيل الدخول
vercel login

# نشر المشروع (للتطوير)
vercel

# نشر للإنتاج
vercel --prod

الطريقة دي أسرع لو انت بتشتغل من التيرمينال. شوف:

  • npm install -g vercel: بيثبّت Vercel CLI عالمياً.
  • vercel login: بيسجّلك دخول.
  • vercel: بينشر المشروع في بيئة التطوير (Preview).
  • vercel --prod: بينشر المشروع للإنتاج.
💡 الفرق بين Preview و Production
Preview هو نسخة تجريبية من الموقع. بتحطها في رابط عشوائي. مفيدة للمعاينة قبل النشر. Production هو الموقع الحقيقي اللي المستخدمين بيشوفوه. لازم تكون متأكد إن كل حاجة شغالة قبل ما تعمل deploy للإنتاج.

إعدادات next.config.js للنشر

next.config.js javascript 
/** @type {import('next').NextConfig} */
const nextConfig = {
  // تحسين الأداء للإنتاج
  experimental: {
    optimizePackageImports: ['lucide-react', '@radix-ui/react-icons']
  },
  
  // إعدادات الصور
  images: {
    remotePatterns: [
      {
        protocol: 'https',
        hostname: '**.supabase.co'
      }
    ],
    formats: ['image/avif', 'image/webp']
  },
  
  // تعطيل Strict Mode في الإنتاج
  reactStrictMode: true,
  
  // تحسين الحزم
  webpack: (config, { isServer }) => {
    if (!isServer) {
      config.resolve.fallback = {
        ...config.resolve.fallback,
        fs: false,
        net: false,
        tls: false
      };
    }
    return config;
  }
};

module.exports = nextConfig;

الإعدادات دي مهمة للإنتاج:

  • optimizePackageImports: بيحسّن المكتبات الكبيرة.
  • images.remotePatterns: بيسمح بالصور من Supabase.
  • reactStrictMode: بيشدد أخطاء React. مفيد للتطوير.
  • webpack fallback: بيعطل بعض المكتبات اللي مش محتاجها على العميل.

🌐 إعداد نطاق مخصص (Custom Domain)

بعد النشر، ستحصل على رابط مثل market.vercel.app. يمكنك تغييره لنطاق مخصص.

خطوات إعداد النطاق المخصص

  1. شراء النطاق من مسجل نطاقات (مثل Namecheap, GoDaddy)
  2. الذهاب إلى Vercel Dashboard → مشروعك → Settings → Domains
  3. إضافة النطاق وكتابته (مثلاً: market.com)
  4. تحديث DNS Records عند مسجل النطاق:
    • Type: A
    • Name: @
    • Value: 76.76.21.21
  5. انتظار التفعيل (قد يستغرق 24-48 ساعة)
DNS Records text 
# للنطاق الرئيسي
Type: A
Name: @
Value: 76.76.21.21

# للنطاق الفرعي (www)
Type: CNAME
Name: www
Value: cname.vercel-dns.com
💡 تثبيت SSL تلقائي
Vercel يوفر شهادة SSL مجانية تلقائياً لكل نطاق. لا حاجة لتثبيت شهادة يدوياً. يعني موقعك هيشتغل على HTTPS من غير أي إعداد إضافي.
⚠️ مهمة بعد تغيير النطاق
بعد ما تغيّر النطاق، لازم تحدّث NEXT_PUBLIC_SITE_URL في متغيرات البيئة على Vercel. وكمان لازم تحدّث روابط Clerk و Supabase عشان تشتغل على النطاق الجديد.

✅ التحقق بعد النشر

بعد النشر، تأكد من فحص جميع الميزات:

الفحص الرابط الملاحظات
الصفحة الرئيسية / تأكد من تحميل الصور والروابط
صفحة المنتجات /products تأكد من عمل البحث والفلتر
تفاصيل المنتج /products/[id] تأكد من عرض الصور والمعلومات
تسجيل الدخول /sign-in تأكد من عمل Clerk
Admin Dashboard /admin تأكد من حماية المسارات
API Routes /api/* تأكد من عمل endpoints
فحص الأداء bash 
# فحص الأداء باستخدام Lighthouse
npx lighthouse https://your-domain.com --output html

# فحص الروابط المكسورة
npx broken-link-checker https://your-domain.com

# فحص SEO
npx seo-checker https://your-domain.com

🔄 النشر التلقائي (CI/CD)

Vercel يوفر النشر التلقائي مع كل push إلى GitHub:

✅ ميزة النشر التلقائي
عند ربط GitHub مع Vercel، سيتم نشر أي تعديل جديد تلقائياً. يعني كل ما تعمل git push، الموقع بيتتحدث. كمان يمكنك إعداد معاينة قبل الدمج (Preview Deployments).
vercel.json (اختياري) json 
{
  "framework": "nextjs",
  "buildCommand": "npm run build",
  "outputDirectory": ".next",
  "installCommand": "npm install",
  "regions": ["iad1"],
  "headers": [
    {
      "source": "/(.*)",
      "headers": [
        {
          "key": "X-Content-Type-Options",
          "value": "nosniff"
        },
        {
          "key": "X-Frame-Options",
          "value": "DENY"
        },
        {
          "key": "X-XSS-Protection",
          "value": "1; mode=block"
        }
      ]
    }
  ]
}

الـ vercel.json ده اختياري. بس لو حطيته، بيديك تحكم أكتر:

  • regions: الموقع هيتنشر في منطقة معينة.iad1 هي أمريكا الشمالية.
  • headers: بيحط headers أمنية. X-Content-Type-Options بيمنع المتصفح من تفسير الملفات بشكل غلط. X-Frame-Options بيمنع الموقع من الظهور في iframe.

📊 مراقبة الأداء

بعد النشر، راقب أداء موقعك باستخدام:

  • Vercel Analytics: مراقبة Web Vitals. بيقيس LCP و FID و CLS.
  • Vercel Speed Insights: قياس سرعة التحميل من أماكن مختلفة.
  • Google Search Console: مراقبة SEO. بيريك الناس بتبحث عن ايه وبتوصل لموقعك ازاي.
  • Uptime Monitoring: مراقبةوقت التشغيل. بيبلغك لو الموقع وقع.
💡 نصيحة مهمة
بعد النشر الأول، راقب الموقع لمدة أسبوع. شوف لو في أخطاء في Console. شوف لو السرعة كويسة. لو لقيت مشاكل، اتصلحها بسرعة قبل ما العملاء يشوفوها.

📝 تمارين عملية

🎯 تمرين 1: نشر المشروع
